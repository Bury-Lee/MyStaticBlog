## 核心观点：ZooKeeper 是一个通用的分布式协调服务

这段话描述的是 ZooKeeper 在数据中心中的**实际部署模式**和**设计哲学**：

### 1. "启动一个 ZooKeeper 集群" —— 共享基础设施

```
┌────────────────────────────────────────┐
│           单个 ZooKeeper 集群           │
│         (5-7 个副本，独立部署)          │
│              ▲ ▲ ▲ ▲ ▲                 │
└──────────────┼─┼─┼─┼─┼─────────────────┘
               │ │ │ │ │
       ┌───────┘ │ │ │ └───────┐
       ▼         ▼ ▼ ▼         ▼
   [Web服务]  [存储系统]  [MapReduce]  [其他服务]
```

**关键洞察**：不是每个服务自己维护一套协调系统，而是**整个数据中心共享一个 ZooKeeper 集群**。

### 2. "通用性使其可用于许多事情" —— 设计哲学

| ZooKeeper 提供的原语 | 不同服务的用法 |
|---------------------|---------------|
| 强一致存储 | 配置管理、主节点状态 |
| 临时节点 + 监听 | 服务发现、主节点选举 |
| 顺序节点 | 分布式队列、任务分配 |
| 原子操作 | 锁、屏障同步 |

### 3. 为什么这样设计？

**经济性**：维护 5-7 个副本的集群成本固定，支撑越多服务，边际成本越低

**可靠性**：专业团队运维一个集群，比每个服务团队自己折腾更稳定

**解耦**：服务只需关心"用"协调服务，不用关心"实现"协调服务

关于Zookeeper的节点

## Zookeeper ZNode 的四种类型
如文中所说,节点就像是一个文件系统,根据这些"文件"的属性分成四种类型

| 类型 | 命名 | 核心特性 |
|------|------|----------|
| **持久节点** | Persistent | 客户端断开后依然存在，需手动删除 |
| **临时节点** | Ephemeral | 客户端会话结束自动删除，**不能拥有子节点** |
| **持久顺序节点** | Persistent_Sequential | 持久节点 + 自动编号后缀（如 `/lock-0000000001`） |
| **临时顺序节点** | Ephemeral_Sequential | 临时节点 + 自动编号后缀，**分布式锁的核心实现** |

## 为什么这么设计？—— 场景驱动

### 1. **持久节点（Persistent）**
**用途**：存储配置信息、元数据、服务注册表等需要长期保留的数据。

```
/zookeeper
/config/database/url  ← 持久保存数据库配置
/services/api-gateway  ← 持久保存服务地址
```

**为什么需要**：分布式系统的配置中心需要数据不因客户端重启而丢失。

---

### 2. **临时节点（Ephemeral）**
**用途**：服务发现、心跳检测、会话绑定。

```
/services/cache-server/node-1  ← 临时节点
# 当 cache-server 进程崩溃或网络断开，节点自动消失/失效/清理
```

**为什么需要**：
- **自动清理机制**：无需担心孤儿节点堆积
- **会话绑定**：节点的生命周期与客户端会话强关联，天然实现"存活检测"

---

### 3. **持久顺序节点（Persistent_Sequential）**
**用途**：全局ID生成器、有序队列、版本控制。

```
/tasks/task-0000000001
/tasks/task-0000000002  ← 严格递增，无冲突
```

**为什么需要**：
- 分布式环境下生成全局唯一且有序的ID非常困难
- Zookeeper用10位数字保证顺序性，解决并发创建时的命名冲突,比如自动生成一个qq号,玩家ID等,此外还有类似场景如订单系统需要按创建顺序处理退款申请:

/tasks/refund-0000000001  ← 用户A的退款申请
/tasks/refund-0000000002  ← 用户B的退款申请  
/tasks/refund-0000000003  ← 用户C的退款申请

工作进程1：取走0000000001，处理完删除
工作进程2：取走0000000002，处理完删除
...

---

### 4. **临时顺序节点（Ephemeral_Sequential）**
**用途**：**分布式锁**（这是最经典的设计）

```
/lock/my-resource/lock-0000000001  ← 客户端A
/lock/my-resource/lock-0000000002  ← 客户端B
/lock/my-resource/lock-0000000003  ← 客户端C
```

**为什么这是好的设计**：

| 问题 | 解决方案 |
|------|----------|
| 锁持有者崩溃？ | 临时特性 → 会话结束自动释放锁 |
| 多个客户端同时抢锁？ | 顺序特性 → 编号最小的获得锁，其余监听前一个节点（避免羊群效应） |
| 如何实现公平锁？ | 天然FIFO顺序 |

**羊群效应（Herd Effect）的避免**：
> 如果所有客户端都监听 `/lock` 节点，释放时所有人都会被唤醒争抢。而顺序节点设计让每个客户端只监听**前一个序号**的节点，形成监听链，极大减少唤醒次数。

## 设计哲学总结

```
┌─────────────────────────────────────────┐
│  持久 vs 临时  →  生命周期管理策略      │
│  顺序 vs 非顺序 →  并发控制与排序需求    │
└─────────────────────────────────────────┘
```

这四种组合覆盖了分布式系统的核心场景：
- **配置管理** → 持久节点
- **服务发现** → 临时节点  
- **全局ID** → 持久顺序节点
- **分布式锁/选举** → 临时顺序节点

Zookeeper通过极简的节点类型设计，支撑了复杂的分布式协调模式，这也是它成为Hadoop、Kafka等系统底层依赖的原因。

Zookeeper的watch机制:本质是监听某个节点的特点变化,在此可以衍生出很多用法,例如:
分布式锁:
传统的分布式锁,几百个节点监听一个锁的释放,那么就会导致锁释放时产生大量的收发请求,这就羊群效应.
而有了watch:
先检查锁的存在,发现有比自己节点小的锁时,设置一个新的预备锁,并watch上一个锁的释放,这样的就可以避免羊群效应,每次只监听一个节点,一个节点也只被一个客户端监听



CRAQ,链式更新是一串的,像是链表(双向)那样,操作会发给头部,头部往下面的节点传递,下面的节点收到后会往上返回ack形成确认链
所以,特点是,上游可能是脏数据,但是尾节点一定有干净数据
如果向脏数据的节点(写入新数据但是未确认)则会向尾部获取干净数据

CRAQ同时提供3种读一致性模型，允许读请求指明可选的一致性类型。
• 强一致性，默认的一致性类型，允许从任意节点均可读取到object最近已提交的数据。
• 最终一致性，从任意节点读，均返回该节点已知的object的最新版本数据（clean）。这可能会导致连续两次从不同节点读取到不一致的数据，因此，不满足单调读一致性保证。
• 具有最大不一致边界的最终一致性，从任意节点能读取到object新写入的未提交的数据（dirty），但需要满足某些限制条件。该模型下，读请求的返回值具有最大不一致的周期（时间或版本号）。对于正常情况下，该不一致表示获取到比最后一次提交的版本更新的版本。对于异常情况下，比如网络分区导致部分节点未参与写请求，该不一致表示获取到比当前已提交的版本更旧的版本。

分布式对象存储系统需要应对各种各样的应用需求，常见的应用场景包括：
• 对于一个object的大部分或者全部的写请求，来源于单一数据中心；
• 一些objects，只与部分数据中心相关；
• 对于热点object，需要更高的副本数；
(应该是适配多读多写的场景)
注:数据中心是像Raft集群那样高可用高容错的分布式存储节点


CRAQ提供一种比较灵活的配置策略，采用two-level命名层级结构，满足多种应用需求。Object identifier由两部分组成：chain identifier和key identifier，其中，chain identifier唯一标识一条复制链，key identifier为复制链内唯一。下面讨论几种配置策略：

## CRAQ 的基本标识结构
- `Object identifier = (chain identifier, key identifier)`
  - `chain identifier`：唯一标识一条复制链（即一组存储节点）。
  - `key identifier`：在该复制链内唯一标识一个 object。
这样可以把不同 object 分配给不同复制链，不同复制链有不同的配置。

### 策略1：`{num_datacenters, chain_size}`
- 只指定**数据中心数量**和**每条链的副本总数**。
- 不明确哪些数据中心参与这条链 → 用**一致性哈希**决定数据中心。
- 缺点：不能保证数据中心在广域链中的顺序。

### 策略2：`{chain_size, dc1, dc2, …, dcN}`
- **明确指定数据中心顺序**（dc1, dc2, …, dcN）。
- 每个数据中心内的**节点数量**由一致哈希根据 `chain identifier` 决定。
- 如果 `chain_size = 0`，表示该数据中心**所有节点**都在链里。
- 链的顺序就是数据中心的顺序：头节点在 dc1，尾节点在 dcN。

### 策略3：`{dc1, chain_size1, …, dcN, chain_sizeN}`
- 与策略2类似，但**每个数据中心可以有不同副本数**。
- 适用于不同数据中心负载/容量不同的场景。

对于是否设置Master的行为:
- **未设置 master**：CRAQ 跨数据中心时，如果发生网络分区，为了安全，要求**多数“节点”**（这里的节点可能是按权重或数据中心数量理解）同意才能继续写。否则降级为只读。
- **设置了 master 数据中心**（例如策略2/3中指定 dc1 为 master）：
  - 正常情况下，**所有写请求只发给 master**（dc1），由 master 协调沿链传递。
  - 临时故障时（比如 dc1 到其他数据中心网络断开），**只有 master 所在分区可以处理写**；
  - 如果 master 本身故障，则由预先指定的下一个数据中心（dc2）临时接管为 master。
  - **重点**：使用 master 模式时，实际上**放弃了跨数据中心的多数派协商**，因为写路径是单一源头 + 链式传播，而不是 quorum 协议。这是一种明确的设计选择——通过牺牲分区容忍下的写入可用性，换取简单性和可预测的延迟。

```md
如果不设置 Master：上海机房（3个节点）和北京机房（3个节点）是平等的。如果光缆断了，上海和北京互相看不见。系统会问：“总共有 6 个节点，我有 3 个（上海方），没达到多数（4个），那我只能只读，不敢写。” —— 这叫多数派协议。

如果设置 Master=上海：你明确告诉系统：“只要上海这个机房还活着，就可以写，北京虽然活着但网络断了也没关系，不等它了。” —— 这叫主从模式。
```

## 单一数据中心内的部署

- 可以用一致哈希部署多条链，但问题：一个节点可能同时是多个链的 HEAD（可能导致写入压力大、状态复杂）。
- 替代方案：用**独立协调服务**（如 ZooKeeper）显式管理链和节点的映射。

---

## 跨数据中心的读优势
CRAQ支持从任意节点读取数据的能力(有助于降低读请求时延)，特别是跨数据中心部署的场景，client可以选择从就近节点（或者负载较低的节点）读取数据。而传统的CR，只有TAIL节点可以接收读请求，很可能需要从远端数据中心读取数据。事实上，根据object访问所呈现的地域属性，可以选择将HEAD节点或者TAIL节点部署在本地数据中心。
(这样类似于一个环,头尾同一,那么就可以同时处理写和读)

通过仔细选择构建广域复制链的数据中心的顺序，可以降低写入时延和减少网络传输开销（一致性哈希方式，无法保证数据中心在广域复制链中的顺序）。相比primary/backup协议并行复制方式，虽然随着数据中心数量增加，写入延时也会明显增加。但是，由于CRAQ支持以pipeline方式处理写请求，可以有效提升写入吞吐量。

---

## 写入延迟与吞吐量的权衡

- 随着数据中心数量增加，写入延迟会线性增加（因为链更长）。
- 但 CRAQ 用 **pipeline** 方式处理写请求（不是等每个副本确认再往下传），所以**吞吐量可以很高**，比 primary/backup 的并行复制在某些场景更好。



传统 Raft：每次写入需要领导者向所有副本发送日志，并等待多数派确认。
代价：网络往返次数多，领导者负担重 → 性能瓶颈明显（高延迟、低吞吐）。
为了解决容错 + 高性能 兼得的问题,CRAQ诞生
解决方案思路 → CRAQ（Chain Replication with Apportioned Queries）
放弃 Raft 的“星型拓扑”（领导者直连所有副本）。
改用链式结构：
写入：头节点 依次向链上传递，尾节点 确认成功 → 只需一次网络传播（单次发送，链上接力）。
读取：通常由尾节点及一部分尾部前面的节点 负责（强一致性），或 CRAQ 允许查询分散到链上节点（提高读性能）。

课程原话:
至于为什么构建这个结构——为什么是这个而不是别的，比如为什么是 Raft？性能原因是：在 Raft 中，如果你记得，如果我们有领导者和一些副本（你知道，一定数量的副本），对吧？我们有一个领导者。它不在链中。这些副本都直接由领导者提供。所以如果一个客户端写入或客户端读取（同样），领导者必须将其发送到每个副本；而在链式复制中，头部只需要做一次发送。这些网络发送实际上相当昂贵。这意味着 Raft 领导者的负载将高于链式复制领导者的负载。这意味着，随着你从客户端获得的每秒客户端请求数增加，Raft 领导者会比链式复制头部更早达到限制并停止变快，因为它做了更多的工作。

这个外部权威消除了分歧的来源：不再有“我们认为”的链，只有“配置中心宣布”的唯一的链。
