> 本文是 **CMU 15-445 Database Systems 第 1 节课**的学习笔记，主题为数据库存储架构与页目录管理。

缓存分级
操作不同层级的内存,需要的时间为:
| 层级             | 延迟量级     | 
| -------------- | -------- |
| CPU寄存器         | < 1ns    | 
| L1/L2/L3 Cache | 1-10ns   | 
| 内存 (RAM)       | 100ns    |
| SSD            | 10-100μs | 
| 磁盘 (HDD)       | 1-10ms   | 
|网络|难以确定|

数据库存储:也是比较普遍的有一个内存池,会把热门的数据页放内存里,如果内存没有才去文件中读取,至于其中的替换算法应该是可以自己决定的

因为其存储本质上还是文件,所以不会被系统特殊对待,对于存储的方式,是通过目录和页的方式,目录类似索引表 / 元数据页,记录着每个页的元数据(这个页里有ID为多少到多少的数据,目前有多少数据等不涉及记录具体信息的数据,以及最关键的数据页存放在哪里),而页则是一部分记录的聚合体.每个页是有固定大小的,可以被设置,一般为4~8KB

一条sql语句的查询是:输入->分析器,分析成具体的查询计划->交给执行器执行(这个有点像Java的把代码编译成字节码然后交给虚拟机执行)

更新数据时会在内存中更改,如果某个页的数据被更改了,那么会被标记为脏页,然后定时或在特定时机写回数据库

对于页ID(Page ID),通过Page ID →（某种元数据/映射结构）→ 磁盘位置（文件 + 偏移）

对于堆文件,可以这样理解,这是申请的文件内存,对于这片内存,如果每次修改数据都要排序,那么代价太高昂了,于是通过一种思想,就是对于堆文件中的记录是运行乱序的,通过数据库自己实现的元数据->实际文件偏移的方法实现映射,这样用起来的话,底层是无序的,但是对于数据库就像是有序的,这样的思想CSAPP里也提到过,空间的申请存储也不会是顺序的,而是看到哪里有空间就插入了,虚拟内存和实际物理内存通过表来实现映射.
但是有的数据库会采用有序文件,例如 InnoDB 的聚簇索引,本身是按主键有序排列的,这种被称之为ISAM,属于是上个世纪被淘汰的技术了

如果是单个文件那么数据页偏移很好计算,就是:第 N 号页面的偏移量 = N × 页面大小
但是当数据很多时,数据库必定会把数据设计为分散于多个文件中.
那么怎么精准定位到一个指定的页(在数据本身是堆文件存储的情况下)?
于是有了页面目录.页面目录本质上是一个类似哈希表的概念性方式,每个数据库的实现都不同,一般是有一个页面目录,记录的应该是目录和页面等所在的文件,会随着写数据页时以安全的方式同步并写入硬盘(持久化),页面目录还会记录空闲空间映射,维护某种数据结构,这个数据结构会记录空闲空间在哪里,类似于CSAPP里的空闲空间表,这样的话就可以快速的查询到空闲空间并写入记录,就不用再全文件扫描了.
不过需要确保操作是故障安全的（failsafe）,因为这个空闲空间映射可能是错误的,因为我们不需要像刷新页目录那样刷新它(为了性能).如果我们跳转到一个我们认为为空但实际并非如此的页，我们需要能够回去找到另一个空白页.
原话:有些数据库实际上会在每页的基础上跟踪该页代表什么，比如它是存储数据，还是存储元数据，等等。所以我们可以查阅它来弄清楚我们在做什么，而不是实际去跳转进去查看内容。

课程上的过程展示:

查询:
1. 查页目录 → 找到文件位置
2. 用简单偏移计算 → 跳到对应页
3. 读取数据

空闲空间映射（FSM）

Postgres 用一个结构叫：
Free Space Map（FSM,空闲表映射）

### 初始状态

有 3 条记录
一个 page 是 8KB
page 0 ≈ 97% 满（其实是"看起来满"，因为最小分配单位是页）

插入 1000 条数据后，所有页的 free space = 0（看起来 100% 满）
但这其实 **不合理**，最后一页不可能刚好填满

### 结论
FSM **没有及时更新**

Postgres 的策略是：
 "我不保证 FSM 实时准确"

因为：

* 更新它有成本
* 偶尔不准也没关系

可能带来的影响：

* 多分配新页（浪费一点空间）
* 但不会出错

## VACUUM 

`VACUUM` 可以理解为：

数据库的"垃圾回收 + 空间整理"

它会：

1. 扫描真实数据页
2. 更新 FSM
3. 标记哪些地方可以复用

---

### 执行 VACUUM 后

你看到：

* 前几页：100% 满
* 最后一页：比如 60% 满

👉 这才是**真实情况**

---

## 删除数据后的行为

当你执行：

* 随机删除一半数据

会发生：

* 页内部出现"空洞"
* FSM 有时会**立即更新**
* 有时不会（取决于策略）

现象：

* 有些页突然有 free space

---

## 再插入数据 + VACUUM

流程：

1. 删除 → 出现空洞
2. 插入 → 尝试复用这些空洞
3. VACUUM → 更新 FSM

结果：

* 空间被重新利用
* FSM 变准确

Postgres是否有压缩功能？
当我检查数据利用率，但页大约只有40%利用率时，有没有办法强制Postgres使用更多空间？基本上就是开始移动数据来填充这些大洞？答案是：是的，`VACUUM` 会做这个，如果你调用 `VACUUM FULL`，它会字面意义上地通过扫描原始表来创建一个新的表副本，然后按顺序写入它们，填充它们，以最小化你使用的页数。但这很昂贵，因为你会加锁，可能锁住整个表，阻止其他任何人更新它。
不同的数据系统会做不同的事情。

对于每个页面都会有一个页眉记录额外的必要信息

如何在一页中存储记录?
初级的方法:记录页内当前有的记录数量,并计算出偏移量,在那里追加数据
但是缺点也是很明显,如果删除了数据,那么就要重新调整才能让它生效

于是有了进阶的方法:槽页
在页眉中后面记录着每条记录开始的偏移量,数据的追加从后面往前写,这样的话会在文件中间相遇,如果把数据删除,就在槽页中把数据标记为删除(如果有机会,有的数据库还会进行排序来把被删除的数据清空释放内存)
