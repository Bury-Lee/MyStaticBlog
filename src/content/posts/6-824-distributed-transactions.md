分布式事务


ACID原则
A - Atomicity（原子性）
定义：如果一个事务包含多个步骤（例如写入多条不同的记录），即使发生失败，也必须保证要么所有写入都成功完成，要么一个都不执行。
意义：绝不能出现因在事务中途发生故障，导致部分更新已可见而部分更新未执行的情况。这就是所谓的“全有或全无”（All or Nothing）。
C - Consistency（一致性）
定义：通常指数据库会强制执行由应用程序声明的某些不变量（Invariants）。
注记：今天我们将暂时不深入探讨这一点。
I - Isolation（隔离性）
定义：这是非常关键的一点。它指的是同时运行的两个事务是否能在事务完成之前看到彼此的更改，或者是否能观察到另一个事务的中间状态更新。
目标：答案是否定的。
技术含义：大多数人所说的“隔离性”通常意味着事务执行必须是可串行化的（Serializable）。稍后我将详细解释这一概念。简而言之，事务之间不能互相看到彼此的更改或中间状态，只能看到完整的事务结果。
D - Durability（持久性）
定义：一旦事务提交（Commit），即数据库向提交该事务的客户端返回“执行成功”的回复后，该事务对数据库的修改必须是持久的。
意义：这些修改不会因某种故障而被擦除。
实践要求：这意味着数据必须被写入非易失性存储（Non-volatile Storage），如磁盘等持久化存储介质。

**可串行化的定义**：
如果存在某种事务执行的顺序（Serial Order），使得该顺序执行产生的结果与实际并发执行的结果相同，那么该并发执行就是可串行化的。

*   **关键点**：实际执行可能包含大量的并行性，但必须产生与某个“一次一个”（One-at-a-time）串行执行相同的结果。
*   **验证方法**：检查并发执行的结果，看是否能找到一种串行执行顺序能复现这些结果。
这些概念在数据库中也有提到

### 3.2 案例分析：银行转账与审计
*   **初始状态**：账户 $x = 10$，账户 $y = 10$。
*   **事务 T1（转账）**：将 1 美元从 $x$ 转移到 $y$。
    *   操作：$x \leftarrow x + 1$, $y \leftarrow y - 1$。
*   **事务 T2（审计）**：读取所有账户余额并打印总和，确保总金额不变。
    *   操作：Read($x$), Read($y$), Print(Sum)。

**可能的串行执行顺序只有两种：**

#### 情况 A：先执行 T1，后执行 T2 (T1 $\to$ T2)
1.  **T1 执行**：$x$ 变为 11，$y$ 变为 9。
2.  **T2 执行**：读取当前的 $x=11$ 和 $y=9$。
3.  **结果**：打印 `11, 9`。

#### 情况 B：先执行 T2，后执行 T1 (T2 $\to$ T1)
1.  **T2 执行**：读取原始的 $x=10$ 和 $y=10$。
2.  **T1 执行**：$x$ 变为 11，$y$ 变为 9（修改生效）。
3.  **结果**：打印 `10, 10`。

**结论**：
*   合法的最终结果只有两种：`11, 9` 或 `10, 10`。
*   如果我们观察到其他任何结果，说明数据库没有提供可串行化执行。
*   **注意**：由于存在多种可能的合法结果（取决于实际调度顺序），程序必须能够接受其中任何一种作为正确结果。随着并发事务数量的增加和复杂度的提升，可能存在的正确结果组合会非常多。

### 3.3 非法执行的示例
假设系统以以下方式交错执行：
1.  T2 开始，读取 $x$（值为 10）。
2.  T1 运行，将 $x$ 加 1（变为 11），将 $y$ 减 1（变为 9）。
3.  T2 继续，读取 $y$（值为 9）。
4.  T2 打印 `10, 9`。

**分析**：结果 `10, 9` 既不是 `11, 9` 也不是 `10, 10`。因此，这种交错执行是不可串行化（Not Serializable）的，是不合法的。


并发控制策略（Concurrency Control Strategies）
为了实现可串行化，我们需要并发控制机制。主要有两大类策略：

悲观并发控制（Pessimistic Concurrency Control）
核心机制：锁（Locking）。
原理：事务在使用任何数据之前(读取和写入)必须先获取该数据的锁。如果其他事务已持有锁，当前事务必须等待。
特点：用性能换取正确性。通过强制串行化顺序来避免冲突。
这里将重点讨论基于**两阶段锁（Two-Phase Locking, 2PL）**的悲观策略。

乐观并发控制（Optimistic Concurrency Control）
核心机制：先执行，后检查。
原理：事务假设不会发生冲突，直接进行读写操作（通常写入临时区域）。在事务结束时，检查是否有其他事务干扰。
若无冲突：提交。
若有冲突：中止（Abort）并重试。
适用场景：当冲突很少发生时，性能优于悲观策略；但当冲突频繁时，会导致大量重试，性能下降。




## 两阶段锁协议（Two-Phase Locking, 2PL）

这是最常见的锁定机制，分为两个明确的阶段：

### 第一阶段：扩展阶段（Growing Phase）
事务在读取或写入任何记录之前，必须先**获取（Acquire）**该记录的锁。
事务可以累积多个锁。

### 第二阶段：收缩阶段（Shrinking Phase）
一旦事务释放了任何一个锁，它就进入了收缩阶段。
**严格规则**：在事务提交（Commit）或中止（Abort）**之前**，它必须持有所有已获取的锁。不允许在中间释放任何锁。

### 为什么必须等到最后才释放锁？
如果允许事务在使用数据后立即释放锁（例如，读取 $x$ 后立即释放），会导致以下问题：

**破坏可串行化**：
    *   假设 T2 读取 $x$ 后释放锁，T1 随即执行并修改 $x$。T2 可能看到不一致的状态或产生非法结果（如之前分析的 `10, 9`）。
**原子性失效（幻读/中间状态可见）**：
    *   假设 T1 修改了 $x$ 后释放锁，但随后因某些原因（如余额不能为负）中止。
    *   如果 T2 在 T1 释放锁之后、T1 完全回滚之前读取了数据，T2 会看到“幽灵值”（Phantom Value），即一个实际上从未存在过的状态。这违反了原子性。
这就是著名的幻读/脏读问题,读取到了不应该存在的("中间态")数据.在数据库中亦有记载

### 死锁（Deadlock）问题
两阶段锁的一个主要副作用是容易引发死锁。
*   **场景**：
    *   T1 锁定 $x$，请求锁定 $y$。
    *   T2 锁定 $y$，请求锁定 $x$。
*   **结果**：双方互相等待，形成循环依赖。
*   **解决**：数据库系统通常通过检测循环（Tracing Cycles）或超时机制来发现死锁，并中止其中一个事务以打破僵局。
<details>
具体的方法有:
1. 集中式死锁检测
核心思想：选一个节点作为全局检测器，所有节点定期将各自的等待图（Wait-For Graph）信息发送给这个中心节点。中心节点合并成全局等待图，判断是否有环。

优点：实现简单，逻辑与单机一致。
缺点：单点故障、性能瓶颈、通信压力大。

2. 分布式死锁检测算法（如路径推动法）
以经典的 Obermarck 算法（路径推动）为例：
每个节点只维护自己本地等待图，并且只关心那些“跨节点”的等待边。
当节点发现一个等待环可能跨越多个节点时，会将这条路径信息（称为“探针”）传递给下一个被等待的节点。
如果某个节点收到的探针又传回了自己，就检测到死锁。
特点：无需中心节点，采用消息传递和环检测。

3. 边追逐法（Chandy-Misra-Haas 算法）
这是分布式死锁检测中非常经典的算法，适用于资源分配系统：
每个事务（或进程）开始时生成一个唯一的ID。
当某个事务等待另一个节点上的资源时，它会发出一个探测消息 (detector, blocker, hop)。
探测消息沿着等待链传播，如果最终探测消息回到发起者（detector = 自己），说明形成了一个环 → 死锁。
关键优化：通过比较ID或跳数来避免重复处理和误报。

4. 超时机制（最简单的“检测”）
很多实际系统（如数据库分布式事务）不是主动检测死锁，而是设置一个锁等待超时。如果一个事务等待超过阈值，就认为可能死锁，主动回滚自己。

优点：无额外通信开销，实现极其简单。
缺点：可能误杀（不是死锁，只是负载高），也可能很久才发现。
</details>

## 原子提交协议（Atomic Commit Protocols）

在分布式环境中，数据分布在多台机器上（例如 Server 1 存 $x$，Server 2 存 $y$）。我们需要确保即使发生部分故障（Partial Failure），整个事务要么在所有节点成功，要么全部失败。这就是**原子提交协议**的作用。

### 6.1 系统架构
*   **事务协调器（Transaction Coordinator）**：驱动整个事务的单一实体，负责发送消息并做出最终决定。
*   **参与者（Participants）**：持有实际数据并执行部分操作的服务器（如 Server A, Server B）。
*   **标识**：每个消息都带有唯一的**事务 ID（TID）**，以便各方跟踪状态。

### 6.2 两阶段提交协议（Two-Phase Commit Protocol, 2PC）

#### 第一阶段：准备阶段（Prepare Phase）
1.  协调器向所有参与者发送 **Prepare（准备）** 消息。
2.  参与者检查自身状态，判断是否能完成事务（例如：是否有死锁、是否已崩溃重启）。
3.  参与者回复 **Yes**（愿意提交）或 **No**（无法提交/必须中止）。
4.  **关键动作**：在发送 Yes 之前，参与者必须将当前的修改日志和锁信息**持久化（Persist）**到磁盘。这是为了应对协调器崩溃的情况。
这个技术也叫预写入(WRL),是在任何事务(包括数据库事务)中应用很广泛的技术

#### 第二阶段：提交或中止阶段（Commit or Abort Phase）
*   **情况 A：所有参与者回复 Yes**
    *   协调器向所有参与者发送 **Commit（提交）** 消息。
    *   参与者执行最终修改，释放锁，并回复确认（Acknowledgment）。
    *   协调器收到所有确认后，可以清理该事务的状态。(持久化)
*   **情况 B：任一参与者回复 No**
    *   协调器向所有参与者发送 **Abort（中止）** 消息。
    *   参与者回滚修改，释放锁。

### 故障处理与持久化挑战

2PC 协议必须处理各种复杂的故障场景，以确保原子性：

#### 场景 1：参与者在准备阶段崩溃
*   **崩溃前发送 Yes**：参与者已持久化了日志。重启后，它读取日志发现自己承诺了提交（Yes），因此必须准备好完成提交，等待协调器的 Commit 消息。
*   **崩溃前未发送 Yes**：重启后无相关记录。若收到协调器的 Prepare 消息，它将回复 No（因为它不知道之前的状态，且不能冒险）。

#### 场景 2：参与者在提交阶段崩溃
*   如果参与者已收到 Commit 消息并持久化了修改，重启后它应继续完成操作并释放锁。
*   如果重复收到 Commit 消息，只要确认事务已持久化，即可再次回复 Ack。

#### 场景 3：协调器崩溃
*   **在发送 Commit 前崩溃**：参与者等待超时。由于没有收到 Commit，它们可以安全地中止（Abort）并释放锁。协调器重启后，若发现日志中无该事务记录，也视为已中止
*   **在发送 Commit 后崩溃**：参与者可能已执行修改但未收到最终确认。此时，参与者不能单方面中止（因为协调器可能已经决定提交）。它们必须**阻塞（Block）**等待，直到协调器重启并读取日志，重新发送 Commit 或 Abort 消息。一定是收到协调者的确认消息才能释放锁
<details>也就是说如果协调器挂了，且参与者已经回复Yes，那参与者会无限期持有锁，直到管理员介入或协调器恢复。
这就是为什么2PC不适合跨数据中心或长事务。</details>

#### 场景 4：网络消息丢失
*   **协调器超时**：若长时间未收到参与者的 Yes/No 回复，协调器可单方面决定中止事务，防止死锁。
*   **参与者超时**：
    *   若处于准备阶段（未发 Yes）：可安全中止。
    *   若已发送 Yes 但未收到 Commit：**必须阻塞等待**。因为此时协调器可能已经收到了所有 Yes 并正在向其他节点发送 Commit。如果此时单方面中止，会导致数据不一致（部分提交）。

### 2PC 的局限性
尽管 2PC 保证了原子性，但它存在显著缺点：
1.  **性能慢**：涉及多轮消息传递和磁盘 I/O（日志写入）。机械硬盘的延迟（如 10ms）会严重限制吞吐量。
2.  **阻塞问题（Blocking）**：在等待协调器决策期间，参与者必须持有锁。如果协调器故障或网络延迟，整个系统可能长时间停滞，影响其他事务。
3.  **可用性低（Low Availability）**：任何关键节点（特别是协调器）的故障都可能导致整个事务挂起。它不适合跨组织或广域网场景。

### 与 Raft 协议的对比
*   **Raft**：旨在实现高可用性（High Availability）。通过复制数据，只要多数派存活即可继续运行。所有节点执行相同操作。
*   **2PC**：旨在保证原子性（Atomicity），牺牲了可用性。协调器是单点决策者，所有参与者必须完成不同任务。
*   **结合方案**：现代系统（如 Google Spanner）尝试结合两者。使用 Raft 或 Paxos 来复制协调器和参与者集群，从而在保持高可用性的同时实现分布式事务。
