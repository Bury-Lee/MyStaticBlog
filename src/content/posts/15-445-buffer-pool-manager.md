# BufferPoolManager 缓存池

## 一、基本概念

首先将**页目录**加载到内存中，然后查询指定的数据页，返回一个对应的指针。该指针不直接指向内存的数据页，而是指向缓存池的某个数据结构（因为数据页可能经常变动，指针实际指向的是数据页的"句柄"）。缓存池管理器负责处理指针与对应数据页的映射关系。

在内存里的数据页称为**帧**。

之前提到过帧的替换算法，此处补充：对于使用情况的追踪是通过一种元数据实现的，类似于引用计数，或使用时间戳记录等（这种技术在 CSAPP 中也有使用过）。

页表需要有并发控制。如果是 API 而非 SQL 使用，客户端在使用完某个数据页后，需要告知数据库使用完毕，减少引用计数。

## 二、Latch（锁存器）与 Lock（锁）的区别

这里使用的是与"锁"不太一样的东西，称为 **Latch（锁存器）**。

- **普通锁（Lock）**：防止"外部逻辑设计/并发/操作错误"，保护内部逻辑和顺序，可能出现死锁问题。
- **Latch**：假设数据库逻辑是完整且没有低级错误的，不会产生死锁，因此 Latch 使用的是一种**互斥锁**。

<details>
<summary>原文摘录</summary>

在系统和数据库世界之间有一个区别：他们是对的，我们是错的。他们会称之为"锁（lock）"，而我们称之为"锁存器（latch）"。正如他所说，它通常是一个互斥量（mutex）。我们稍后会覆盖锁，但我们不能使用"锁"这个词，因为在数据库世界中，"锁"意味着保护数据库内部的高层概念或逻辑实体。我可以对表加锁、对元组加锁、对索引条目加锁，并且数据系统可以提供额外的保护机制，以确保如果发生死锁，会杀死持有锁的线程或工作者之一来打破死锁。锁是处理外部世界的，并在数据库上执行操作，因此我们必须假设它们是愚蠢的，必须确保它们不会锁死整个系统。

锁存器是我们作为数据库系统开发人员使用的内部保护机制，用于保护关键部分。因为我们是有偿的来构建这个系统，所以我们不能愚蠢行事，因此我们有责任确保没有死锁——不会有更上层的东西杀死我们的死锁同时也不会产生死锁。这是通过程序员纪律来确保我们不犯傻的。所以在这种情况下，是的，你可以使用低级互斥量，可以使用标准库中的互斥量（在 Linux 中实际上是几个文本），我们稍后会覆盖。但大多数数据系统不想使用任何已经简化你的东西。我们将自己实现自己的互斥量，自己实现自己的锁存器。我们稍后会覆盖如何做到这一点。当我们说"锁存器"时，想想像保护关键部分的互斥量，因此我们必须确保不要产生死锁。

下一个问题你可能会有："锁存器和页表与页目录有什么区别？"记住，页目录是磁盘上的那个东西，它将页面 ID 映射到数据库文件或文件中的某个页面位置。每次我们添加更多页面到我们的表中或更新任何东西时，我们必须更新该页目录。如果我们重启系统，我们会重新加载页目录，并且需要与实际文件本身同步。

</details>

## 三、页表（Page Table）与页目录（Page Directory）的区别

- **页目录**：磁盘上的结构，将页面 ID 映射到数据库文件或文件中的某个页面位置。每次添加更多页面到表中或更新任何内容时，必须更新页目录。重启系统时会重新加载页目录，并需要与实际文件本身同步。

## 四、mmap 与虚拟内存

**mmap** 是内存映射。假设有一些磁盘文件，调用并打开它们进行映射，将文件的内容映射到内存中供应用程序使用。它会**懒加载（lazily）**，不会将所有文件内容都加载到内存中，而是给定起点和终点（基于大小），然后说："好的，这是你可以访问该数据的内存区域。"如果你尝试访问尚未在内存中的内容，就会发生**缺页异常（page fault）**（注：这里的页是内存页而不是数据页，要注意区分），操作系统会将其读回。

<details>
<summary>原文摘录</summary>

所以又回到了我们的数据库文件：这里有一个虚拟内存的东西，什么都没有填充。我的线程说："我要访问第 1 页。"第 1 页不在物理内存中，所以会发生缺页异常。当这个进程发生时，操作系统复制该页面并将其放入内存，然后更新虚拟内存以指向现在的第 1 页。同样的事情：如果我想访问第 3 页，复制它进来就完成了。

现在如果我尝试访问另一个页面，比如第 2 页，我没有它。操作系统识别出我没有更多物理内存了，所以再次发生缺页异常，说："好的，你想要的东西在内存中。"于是它决定要驱逐哪些东西。当它这样做时，它会暂停你的线程、暂停你的进程，因为它试图对你透明，让你不知道"哦，你想要的东西不在内存中"。它只是停止你运行，重新调度你，以便将所需的东西带入内存。一旦它在内存中，你就获得指向它的指针，然后对想要的东西做任何事。

</details>

## 五、为什么不用系统 API，而自己实现缓存管理器？

因为系统做这些事情时会**暂停线程**，还存在其他不协调行为（如刷回时机不可控、替换策略不可控、API 过于简陋、适用性不高）。对于数据库查询来说，数据库与操作系统的协作太差了。

<details>
<summary>原文摘录</summary>

**Prof.** 操作系统显然在隐藏决定如何在磁盘和内存之间移动东西的机制，给你那种拥有比实际更多内存的错觉。如果我说它会暂停你的线程，那很糟糕，对吧？如果我运行我的查询，在我的系统中找不到一个线程，那么我只能一次运行一个线程，它将是一个长时间的停顿，因为它要去内存外获取东西并带回磁盘。

好的，那如果我有多个线程同时运行呢？这会不会隐藏一些停顿，因为一个线程可以在另一个线程停顿的同时继续运行？这就是我们的问题之一。我想开场幻灯片上说我要"谋杀"你的数据库系统。所以你们大家都认为这是个坏主意，但你们可能还不知道为什么。

好的，以下是它糟糕的原因。第一点是：操作系统可以随意在任何时候一次性刷新任何脏页。我在开头说过，这是个好主意吗？不是。如果我向页面写入数据（比如更新查询更新一个页面），如果我在单个查询中更新多个页面，我会确保所有更改都已原子性地写出，然后我说"此操作成功完成"。但正如我之前所说，我们不能保证硬件一次只能写入超过 4KB 的页面。所以如果我正在使一堆不同的页面变脏，我可能还在等待决定这是否真的会正式完成，而操作系统说："好的，我需要空间了，让我开始刷新你的脏页。"然后现在如果我崩溃并回来，我得弄清楚整个操作系统在我崩溃前到底写了什么，而我不知道它写了什么，因为它是在我看不见的情况下为我做的所有事情。那太糟糕了。

我们已经谈到了停顿：显然，操作系统不知道哪些页面在内存中，因为它是为我们管理所有这些的。你不能查看页表并对操作系统说："现在实际上在内存中的是什么？"你尝试触摸某物，然后最终会停顿。

你怎么绕过这个问题呢？你可以有一个东西偶尔"触碰"（tickles）页面，由一个单独的线程执行。如果它停顿了也没关系，这样当查询尝试去触摸该页面时就不会停顿。但现在你要构建很多额外的基础设施来克服操作系统的问题，你最终会构建很多东西，而你需要任何人都有一个缓冲池管理器。此外，你怎么知道要触碰哪些页面？怎么知道在你需要之前想要触碰哪些页面？如果你还没有运行查询呢？那很糟糕。

如果现在发生错误呢？你会得到异常吗？你会得到一个错误号吗？不，你会得到中断（interrupt）。你如何在你的程序中处理中断？中断处理器在哪里？你必须把它放在每个地方，因为现在在系统的任何部分，当你访问内存但由 mmap 支持时，你也必须包括中断处理器，因为你突然尝试触摸某物并收到 SIGBUS，你必须决定如何处理这个错误。至少如果你使用 libc 或 POSIX API（如 f-read, f-write），你会得到一个错误号，但它有时会欺骗你。我们稍后会提到这一点，但至少现在你知道你在访问磁盘上的某物，我把它带到内存中，然后我的系统其余部分可以在内存中操作这些页面，我不担心它们被某种方式交换出去，或者操作系统发生的任何事件都隐藏在我之外。

然后我们不会真正讨论性能问题，但请相信我，它会很慢。操作系统如何维护那个映射？他们基本上有自己的页表版本。他们会用什么来保护它吗？是互斥锁。那他们都互斥锁好吗？不，是 libc 的 futex，我认为有更好的东西，即使 Linus 不这么觉得。所以现在我们要做的所有事情——稍后我们会看到这些优化——减少页表上的竞争，聪明地决定驱逐什么、为什么驱逐以及何时驱逐，都被抛到九霄云外了，因为操作系统为我们做了一切。

## 六、替换算法：Clock（时钟算法）

LRU 有一个改进：**Clock Algorithm（时钟算法）**，也称为 **Second Chance**。这是 Linux 使用的方法（Linux 使用多手钟，但想法基本相同）。

这是一种对 LRU 的近似：
- 不需要记录时间戳，只需为每个内存页跟踪一个 bit，表示自上次检查以来是否被访问过。
- 每次需要驱逐某页时：
  - 如果该位为 1 → 表示已被访问，将其重置为 0。
  - 如果该位为 0 → 表示自上次检查以来未被访问，可以直接丢弃并填入新页面，将其位设为 0。
- 继续循环，最后时钟可能从头开始。

<details>
<summary>原文示例</summary>

**Stud.** 当我驱逐第 2 页并带入第 5 页时，我说它的引用位（ref bit）是 0 而不是 1。这里的想法是：我带入了它，但我没有历史知道它是否会被再次访问，所以我将其设为 0。也许我必须带它进来是为了服务需要它的请求。这是可以的，因为我仍然在这些东西上维护固定（pins），但我将其设为 0，这样如果它是一次性读取，它就会被驱逐；如果有人真正需要它，他的位将被设为 1，因此不会被驱逐。

**Prof.** 是的。问题是：如果你检查时它们都设为 1，你会怎么做？

**Stud.** 如果你都是 1，你会做什么？

**Prof.** 当你检查并发现是 1 时，你将其设为 0，绕回来，然后被驱逐。在那一点上，如果你陷入无限循环，基本上设置一个计数器，记录我看了多少次。如果因为我无法驱逐任何东西（因为它们都被固定且位都是 1），你就拒绝查询，说你无法运行它，抛出内存不足错误之类的。但在那种情况下，查询已经慢得离谱了，人类或操作员会知道系统过载并采取其他措施缓解。

</details>

## 七、顺序洪水（Sequential Flooding）问题

LRU 和 Clock 实现简单，但容易受到数据系统中称为**全表扫描污染（scan pollution）**的问题影响。

如果一个查询到来并想要对整个表进行全表扫描（sequential scan），它将读取表中的每一页一次或多次。这会破坏缓冲池及其跟踪元数据的方式，因为现在所有东西看起来都是最近被访问的，它会丢弃那些实际应该保留的东西，只保留最近被访问的东西。

换个角度理解：这相当于把文件全部流式地加载进内存一次，所以显然会很慢。

<details>
<summary>延伸讨论（LRU-K 等）</summary>

在 OLAP 工作负载或分析查询中，有时最近使用的页面正是你想要驱逐的页面：扫描一次，带入页面，读取它，然后在这个查询中永远不会再读该页面，也许这是一次性的事情，没人想再读。那实际上是我想要驱逐的——最近使用的，而不是最近最少使用的。

问题在于 LRU 和 Clock 基本上存储少量信息（如最后访问时间），无法告诉你过去访问了多少次，所以我们处于不利地位，无法决定是否应该保留它在内存中。

克服这个问题的方法是 **LRU-K（Least Recently Used with K references）**，来自发明 LSM Tree 的同一位作者的论文（1990 年代）。核心思想：只保留少量额外元数据，跟踪页面被访问的最后 K 次时间戳。当需要决定驱逐时，查看这段历史，找出从上次访问到下一次访问时间间隔最长的页面，那就是要驱逐的。

SQL Server 做了额外优化：跟踪页面是否在同一事务中被重复访问，避免将多次访问误判为高热。

</details>

## 八、冷启动（Cold Starts）

如果停止系统并重新启动，LRU 的元数据会丢失。解决方案：维护一个小缓存，记录页面被驱逐前在内存中的访问频率，这样重新加载后算法可以快速收敛。**Postgres 采用了这种方式**。

## 九、MySQL 的近似 LRU-K（两段式 LRU）

MySQL 使用称为**近似 LRU-K（approximate LRU-K）**的技术：

- 仍然是一个页面访问顺序的链表，但有两个入口点（两个头部）：
  - **年轻部分（Young Section）**
  - **旧部分（Old Section）**
- 页面第一次被访问时，不添加到 LRU 链开头，而是添加到**旧部分的头部**（中点插入策略）。
- 如果该页面再次被访问，则从旧部分抽出，移到**年轻部分的头部**。
- 驱逐总是从**旧部分的尾部**进行。

> 图片占位：`![alt text](image-6.png)`

这种设计本质上近似于 LRU-2（K=2）：只有被访问至少两次的页面才能进入年轻部分，获得长期保留资格。

## 十、Postgres 的局部替换策略（环形缓冲区）

### 问题背景
全局 LRU 策略下，全表扫描会把大量页面读入缓冲池，即使这些页面只使用一次，也会冲走其他查询的热点页面。

### Postgres 解决方案
- **环形缓冲区（Ring Buffer）**：为每个全表扫描查询分配一个独立的、较小的缓冲池区域（默认约 200~256 KB，对应 32 个 8KB 页面）。
- **替换策略隔离**：该查询读入页面时，只在这个环形缓冲区内部进行局部 LRU；当空间用完，仅驱逐这个区域内最久未使用的页面，完全不碰缓冲池其他部分。
- **访问共享**：其他查询仍可访问环形缓冲区中缓存的页面（通过页号映射到帧），但驱逐决策仅参考本查询的访问序列。

> **关键点**：这是"按查询"的局部替换，而非全局。这是 Postgres 和 MySQL/InnoDB 的重要区别之一。

## 十一、优先级提示（Priority Hints）与软固定（Soft Pin）

### 动机
简单 LRU 系列策略依赖**过去的访问模式**预测未来，但有些页面**结构上就是重要的**（例如 B+ 树的根节点），即使当前未被频繁访问，未来也可能成为瓶颈。

### 软固定（Soft Pin）
- **硬固定（Pin）**：页面正在被使用，不能驱逐（硬约束）。
- **软固定**：提示替换策略"这个页面很重要，如果可能，不要驱逐"，但若内存压力极大，驱逐也是允许的。

### 典型应用场景：B+ 树索引
- **根页面**：每次插入/查找都必须访问。
- **连续插入（单调递增键）**：所有插入都走向最右侧叶节点，路径上的内节点（尤其是根）会被反复命中。
- **提示内容**："这些页面非常重要，请尽量保留在年轻部分。"

### 硬固定 vs 软固定

| 类型 | 是否绝对禁止驱逐 | 使用场景 |
|------|----------------|----------|
| 硬固定 | 是 | 正在读/写的事务，必须禁止驱逐 |
| 软固定 | 否（尽量保留） | 提示重要性，优化策略 |

## 十二、三种策略对比总结

| 特性 | 两段式 LRU (MySQL/SQL Server) | Postgres 环形缓冲区 | 优先级提示（软固定） |
|------|------------------------------|--------------------|---------------------|
| **范围** | 全局缓冲池 | 按查询的局部区域 | 可全局或按页面设置 |
| **防止扫描污染** | 有效（中点插入） | 完全隔离 | 不直接解决，需手动提示 |
| **页面升级条件** | 第二次访问（进入年轻部分） | 不升级，仅在环形区内 LRU | 不依赖升级，直接影响保留倾向 |
| **元数据开销** | 低（仅链表+标志位） | 低（每个查询独立小块） | 极低（一个提示标记） |
| **典型数据库** | InnoDB, SQL Server | PostgreSQL | 理论上均可扩展 |
| **对索引根节点的保护** | 依靠多次访问自动进入年轻部分 | 无特殊保护（除非通过提示+扩展） | 显式保留提示 |

## 十三、思考与延伸

1. **为什么 Postgres 不默认使用两段式 LRU？**
   - 两段式 LRU 依然会在全局竞争，而 Postgres 的环形缓冲区更彻底地实现了"大查询不计入全局 LRU 历史"。

2. **软固定的实际接口**
   - 标准 SQL 中没有优先级提示，但某些数据库（如 Oracle 的 `KEEP` 池）提供了类似功能。
   - 学术界研究如 **Hints for buffer replacement**，可传递"顺序扫描"、"随机访问"、"重要节点"等标签。

3. **LRU-K vs 两段式 LRU**
   - LRU-K 需要记录 K 次访问时间戳，元数据较多；两段式 LRU 是工程化的折中，只区分"访问 1 次"和"访问 ≥2 次"。

4. **未来趋势**：自适应替换（如 ARC、LIRS）、机器学习预测访问模式、基于查询计划的显式内存预留。

## 十四、脏页驱逐成本与后台写入器（Background Writer）

### 理想快速路径
当需要驱逐一个页面以重用帧时：
- 页面是**干净的（未修改）** → 直接丢弃，帧清零，立即复用。成本极低，仅为内存操作。

### 脏页的代价
页面可能是**脏的（已修改）**：
- 必须先写回磁盘（flush），确保持久化，然后才能复用该帧。
- 写盘是**同步 I/O**，会阻塞当前事务/查询，延迟显著增大。

### 后台写入器（Background Writer）

**基本设计思想**：
- 后台独立线程（或周期任务）主动扫描缓冲池，找到脏页，提前写回磁盘。
- 写回后，页面在内存中仍然是干净的（保留缓存），只是脏标志被清除。
- 当驱逐策略真正需要这个帧时，它已经是干净页面 → 走快速路径，无需阻塞。

**时间线拆解**：
- 没有后台写入器：页面变脏 → 一直脏 → 驱逐时被迫写盘（同步阻塞）。
- 有后台写入器：页面变脏 → 后台写入器在"空闲时段"悄悄写盘 → 页面变干净 → 驱逐时直接丢弃。

**重要区别**：脏页写出 ≠ 驱逐。写出后页面仍在内存中，继续服务读请求。

### 后台写入器的调度策略

**激进 vs 保守**：
- 激进：高频刷脏 → 几乎总有干净页可驱逐 → 查询延迟稳定，但后台 I/O 占用资源。
- 保守：几乎不刷脏 → 查询自己承担写脏页的阻塞，可能偶尔卡顿。

各数据库暴露不同参数（如 InnoDB 的 `innodb_max_dirty_pages_pct`、`innodb_io_capacity` 等）。

**更聪明的做法**：
利用替换策略的信息，优先刷那些"替换策略认为很快会被驱逐的脏页"。例如，LRU 尾部的脏页先刷，因为它们大概率是下一个被选中者。

## 十五、操作系统的局限（为什么数据库要自己做）

### 1. I/O 调度
操作系统可以合并、重排序 I/O 请求，但不知道请求之间的逻辑关系，也不知道事务的恢复顺序。

### 2. 缺失的语义
数据库有 **WAL（预写日志）**：必须先写日志，后写数据页。操作系统可能把脏数据页写回磁盘，但对应的日志还没落盘 → 崩溃后无法恢复。

### 3. I/O 优先级限制
Linux 可以在进程级别用 `ionice` 设置优先级，但不能对单个 I/O 请求精细标记。数据库无法告诉操作系统："这个读请求是用户事务的，优先级高；那个预读请求是背景的，优先级低。"

### 4. 自己实现内存管理的优势
- 自定义内部队列与请求排序策略：关键查询优先、后台清理延后、先服务持有大量锁的查询。
- 异步 I/O 与 **io_uring**：线程不阻塞等待，注册请求，完成后通知。
- 直接 I/O（`O_DIRECT`）：绕过 OS 页缓存，只有一份副本，完全由数据系统控制缓存。

> **反面例子**：PostgreSQL 至今依赖 OS 页缓存，仅将 25-30% 内存分配给自己的缓冲池（其他系统用 80%），导致页被驱逐后仍残留在 OS 页缓存中。正在尝试引入直接 I/O。

### 5. fsync 的陷阱
`fwrite` 仅将数据复制到内核缓冲区，崩溃可能丢失数据 → 需要 `fsync` 强制刷盘。  
Linux 上 `fsync` 有一个持续 20 年的 bug：若 `fsync` 失败，Linux 会将脏页标记为干净；第二次调用 `fsync` 会返回成功，但数据实际并未持久化。影响 PostgreSQL、MySQL、MongoDB 等（2018 年曝光）。

**结论**：操作系统不是朋友，应尽可能自己管理一切。

## 十六、缓冲池管理器的优化技术

### 6.1 多缓冲池实例
- 按表、索引、表空间或逻辑数据库划分内存池，减少竞争。
- 映射方式：利用记录 ID 中的对象 ID 定位到对应缓存池，或简单哈希取模（MySQL 做法）。
- 每个实例可独立配置页面大小、替换策略（如 LRU-K）、优先级。
- **限制**：同一个页不能出现在不同的缓存池里，因为跨缓存池协调很困难。

### 6.2 预取（Prefetching）
- 基于查询计划提前识别需要访问的页面。
- **顺序预取**：全表扫描时提前读取后续页面（OS 也能做）。
- **索引预取**：利用 B+ 树叶子节点的兄弟指针，预取非连续但逻辑上需要的页面（OS 无法做到，因为不理解数据结构）。

### 6.3 扫描共享（Synchronized Scans）
- 多个并发查询访问相同页面时，让它们"搭便车"共享同一个扫描游标。
- 避免独立扫描导致页反复被驱逐和重读。
- 支持：SQL Server、DB2、Teradata、PostgreSQL（Oracle 仅限完全相同的 SQL 字符串）。

### 6.4 缓冲池绕过（Buffer Pool Bypass）/ 轻量扫描
- 为临时数据或一次性查询分配私有小缓冲区，避免污染主缓冲池的驱逐元数据。
- 读取页面放入私有区，用完后直接丢弃，不更新全局页表。
- 企业级系统支持，开源系统较少见。

## 十七、操作系统 vs. 数据系统：核心对比

| | 操作系统 | 数据系统 |
|--|---------|----------|
| 查询计划 | 不知道 | 知道 |
| 驱逐决策 | 基于缺页历史 | 基于语义（表、索引、锁、事务） |
| 预取 | 顺序预取 | 顺序预取 + 索引预取 |
| I/O 优先级 | 进程级 | 请求级（可自定义） |
| 恢复语义 | 不支持 WAL 顺序保证 | 支持 |
