# Google Spanner 论文笔记

## 为什么选这篇论文

Spanner 是一个罕见的、能在**全球分散的数据中心**之间提供分布式事务的系统。数据可能散布在北京到上海、甚至中国到美国这样遥远的地方。在生产系统中，这种情况几乎从未出现过。

### 设计动机

Google 内部（尤其是广告系统）存在很多大型数据库，数据被手动分片到大量 MySQL 和 Bigtable 中：

| 痛点 | 说明 |
|------|------|
| 维护笨拙 | 分片过程手动且耗时 |
| 事务受限 | 无法跨多个服务器执行事务 |

**目标**：让数据更广泛地分布，同时支持跨分片事务。

---

## 正确性要求

> 来自课程原文

Spanner 追求两层正确性：

### 1. 可串行化（Serializability）
- 事务必须按某种串行顺序执行
- 只读事务不应看到读写事务的部分写入
- 读写事务的写入要么全部可见，要么全部不可见

### 2. 外部一致性（External Consistency）
- **如果事务 T2 在 T1 提交之后才开始，则 T2 必须看到 T1 的写入**
- 本质 = 可串行化 + 实时约束
- 与"线性一致性"等价，只是作用在**事务级别**而非单个读写级别
- 对程序员非常友好，是一个极强的一致性属性

## 整体架构

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  数据中心 A  │    │  数据中心 B  │    │  数据中心 C  │
│  ┌────────┐ │    │  ┌────────┐ │    │  ┌────────┐ │
│  │ 副本 1  │◄├────┼─►│ 副本 2 │◄├───┼─►│ 副本 3  │ │
│  └────────┘ │    │  └────────┘ │    │  └────────┘ │
│     Paxos   │    │     Paxos   │    │     Paxos   │
│     组 A    │    │     组 A    │    │     组 A    │
└─────────────┘    └─────────────┘    └─────────────┘
       ▲                  ▲                  ▲
       └──────────────────┼──────────────────┘
                          │
                    ┌─────┴─────┐
                    │  客户端    │
                    │ (就近访问) │
                    └───────────┘
```

- 数据跨多个数据中心复制
- 每个表/键范围 = 一个 **Paxos 组**（类似 Raft），实现高可用与容错
- 不同表形成不同 Paxos 组，支持并行操作
- 客户端就近访问本地副本，降低延迟

## 读写事务：2PL + 2PC

### 核心机制

使用 **两阶段锁 + 两阶段提交**

### 执行流程

```
客户端(事务管理器)
     │
     │  1. 读数据 + 加锁
     ▼
┌────────────┐    ┌────────────┐
│ 分片领导者A │    │ 分片领导者B │
│  (维护锁表) │    │  (维护锁表) │
│   Paxos组  │    │   Paxos组  │
└────────────┘    └────────────┘
     │                  │
     │  2. 执行写操作     │
     │  (如 X-1, Y+1)   │
     │                  │
     └────────┬─────────┘
              │
              ▼
     ┌────────────────┐
     │   事务协调器     │
     │  (也是Paxos组)  │
     │  运行 2PC 协议   │
     └────────────────┘
              │
     ┌────────┼────────┐
     ▼        ▼        ▼
  准备请求  准备请求  准备请求
     │        │        │
     ▼        ▼        ▼
  各分片通过Paxos写入"已准备"状态
     │        │        │
     └────────┼────────┘
              ▼
     协调器收到全部"OK"
              │
              ▼
   通过 Paxos 写入"已提交" ← **提交点**
              │
              ▼
      通知参与者释放锁
```

### 特点

| 特性 | 说明 |
|------|------|
| Paxos 化 | 参与者与协调器都是 Paxos 组，**避免 2PC 阻塞问题** |
| 锁表不复制 | 提升读取性能，但牺牲部分容错 |
| 延迟较高 | 读写事务延迟 ~ 几百ms，需跨数据中心通信 |

## 只读事务优化

### 需求背景

> 广告系统以只读事务为主——数十亿次只读 vs 数百万次读写。

三个看似矛盾的需求：
- ⚡ **高性能**：读写事务 ~ 100ms，只读事务 ~ 5-10ms
- 🔒 **强一致性**：外部一致性 + 可串行化
- 📖 **读不阻塞写、写不阻塞读**

### 直观但错误的方法

每次读取最新值 → **会破坏可串行化**，可能出现幻读和脏读。

---

## 核心方案：快照隔离 + 多版本

### 基础思想：为每个值带上时间戳

```
Key: balance
├── value: 100 @ timestamp=10  ← 旧版本
├── value:  90 @ timestamp=15  ← 旧版本
└── value:  80 @ timestamp=20  ← 当前版本
```

- 读写事务开始时选择一个**读快照时间戳**，所有读基于这个快照
- 写操作先缓存，提交时获得一个**提交时间戳** ( ≥ 读时间戳 )，然后写入新版本
- 这称为 **多版本并发控制（MVCC）**

### 读不阻塞写，写不阻塞读

```
       时间轴 ──────────────────────────────►
       
读写事务T1:    [快照@15]──读──写(缓存)──[提交@20]──写入版本20
只读事务T2:          [快照@18]──读版本15──返回  ← 不受T1影响
只读事务T3:                            [快照@22]──读版本20──返回
```

## 外部一致性的直观理解

> 我的理解

事务被赋予一个**提交时间戳**，这个时间戳在逻辑上代表事务"似乎发生"的那个瞬间。

尽管物理上准备、提交、复制需要时间，但从外部观察者角度看：

```
效果等同于：
  事务在它的时间戳瞬间原子地生效
  ├── 更小时间戳的所有操作 "发生在它之前"
  └── 更大时间戳的所有操作 "发生在它之后"
```

这就是 **外部一致性**（= 线性一致性 + 因果一致性）。

## 从副本读取的难题

> 学生提问：如果读的是本地副本，而副本还没有收到最新数据，怎么保证不读到过时数据？

### Spanner 的解决之道：Safe Time

```
副本的 Paxos 日志：
┌─────┬─────┬─────┬─────┬─────┐
│ T=8 │ T=10│ T=12│ T=14│ ... │
└─────┴─────┴─────┴─────┴─────┘
                          ▲
                    Safe Time = 14

读取请求 T=15：
  → Safe Time(14) < 15
  → 必须等待，直到收到 ≥15 的写入
  → 等待...
  → 收到 T=16 的写入后，Safe Time→16
  → 现在可以安全返回 T=15 时的数据 ✓
```

### 核心规则

在执行时间戳 `T_read` 的读取前，副本必须：

1. **等待一个时间戳 > `T_read` 的写入**（确认没有更早的未到达写入）
2. **等待所有时间戳 ≤ `T_read` 的准备中事务完成**（已准备但未提交的事务，必须确认其提交或回滚）

> 对于繁忙的系统，写入持续到来，等待时间几乎为零。

---

## 时钟问题

### 时间戳太大 → 等待更久

```
实际应该读 T=15，却拿到了 T=25
  → 必须等待，直到副本收到 >25 的写入
  → 只是延迟，不会出错 ✓
```

### 时间戳太小 → 读取到旧数据（错误！）

```
T3 实际在时间 10 开始，但时钟只给了 9
  → T3 基于时间戳 9 读取快照
  → 看到的是旧数据
  → 从外部看像是"时间倒流" ✗
```

---

## TrueTime：解决时钟不可靠问题

### TrueTime API

不返回精确值，而是返回一个**时间区间**：

```
请求当前时间
    │
    ▼
TT.now() = [12:00:00.000, 12:00:00.007]
           └─ earliest ─┘ └─ latest ─┘
           
  真实时间一定落在这个区间内
```

> 基于 **GPS + 原子钟**实现，误差通常 < 7ms

### 两条关键规则

#### 开始规则（Start Rule）
> 事务时间戳 = 当前 TrueTime 区间的 **Latest** 值

```
TT.now() = [10.000, 10.007]
事务时间戳 = 10.007  ← 取"未来"端
```
保证该时间戳尚未被所有机器确认已过去。

#### 提交等待规则（Commit Wait Rule）

> 只在读写事务中生效

```
决定提交 @ ts = 10.007
            │
            │  等待...
            ▼
   直到 TT.now().earliest > 10.007
   （确保"提交时刻"已经成为过去）
            │
            ▼
        通知客户端成功
```

**目标**：保证事务提交时，它的"时间"已经被所有人确认属于过去。

### Safe Time 机制

每个副本维护自己的 **Safe Time**：

```
可以安全响应 T_read 的条件：
  ├─ 本地 Paxos 日志的最新时间戳 ≥ T_read
  └─ 所有 ≤ T_read 的准备中事务已确定结果
```

---

## 总结

| 机制 | 作用 |
|------|------|
| **Paxos 组** | 高可用，避免 2PC 阻塞 |
| **MVCC + 快照隔离** | 读写不互锁，只读事务极快 |
| **时间戳 = 全局顺序** | 实现外部一致性 |
| **Safe Time** | 允许读本地副本，同时保证正确性 |
| **TrueTime** | 用物理手段约束时钟误差，使时间戳有意义 |
| **Commit Wait** | 保证事务的"提交时刻"成为真实过去 |

Spanner 用一套**硬件 + 协议 + 等待规则的组合**，在分布式系统中实现了看似不可能的"全球强一致性事务"。
